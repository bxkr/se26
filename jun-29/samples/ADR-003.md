# ADR-003: Кэширование ответов API прогнозов

**Статус:** Принят  
**Дата:** 29.06.2026  
**Связано с требованием:** NFR latency (p99 < 300 мс для API аналитики), US-1 «Аналитик смотрит ошибку прогноза», US-3 «Пользователь смотрит прогноз», US-6 «Партнёры получают прогнозы массивами».

### Контекст

Прогноз для станции пересчитывается редко (батчевый процесс, не чаще 1 раза в час), но читается часто: аналитики, пользователи сайта и партнёры запрашивают одни и те же данные независимо. Без кэша каждый запрос идёт в БД; при одновременных запросах от аналитиков и партнёров нагрузка на БД будет кратно выше необходимой, и p99 < 300 мс становится трудновыполнимым.

### Решение

Кэшируем ответы прогнозов в Redis с TTL = 10 мин. При пересчёте прогноза сервис прогнозирования инвалидирует соответствующие ключи. Ключ кэша: `forecast:{station_id}:{date}`.

### Рассмотренные альтернативы

- **In-process кэш (в памяти сервиса)** — отвергнуто: при нескольких репликах сервиса кэши не синхронизированы → неконсистентные ответы; при перезапуске кэш сбрасывается.
- **CDN-кэширование** — отвергнуто: прогнозы персонализированы по станции и периоду, часть запросов авторизована (партнёры); CDN плохо подходит для мелкогранулярной инвалидации.
- **Без кэша (прямые запросы в БД)** — отвергнуто: не гарантирует p99 < 300 мс при конкурентной нагрузке от нескольких категорий пользователей.

### Последствия

- **Плюсы:** снижает нагрузку на БД; ускоряет повторные запросы до <5 мс; горизонтально масштабируется.
- **Минусы / цена:** риск показа устаревшего прогноза в пределах TTL (до 10 мин); необходима логика инвалидации при пересчёте; Redis — дополнительный компонент инфраструктуры с требованиями к отказоустойчивости.

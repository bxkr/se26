## Шаг 1. Роли (акторы)

Кто пользуется системой или взаимодействует с ней.

| Роль                   | Что хочет от системы                   |
|------------------------|----------------------------------------|
| Аналитик               | смотреть ошибку прогноза по станциям   |
| Станция-сенсор         | сдавать поток измерений факта          |
| Пользователь           | смотреть прогноз на сайте              |
| Администратор          | добавлять/изменять источники измерений |
| Оператор               | следит за состоянием сервера и БД      |
| Пресса                 | получать сводки                        |
| API-клиенты (партнеры) | получают данные массивами              |

## Шаг 2. User stories
 
- **US-1.** Как аналитик, я хочу смотреть ошибку прогноза по станциям, чтобы выявлять несовпадения с фактической погодой, анализировать и сообщать администраторам.
  - **Дано**: старый прогноз на текущий день. **Когда**: аналитик запросил прогноз. **Тогда**: система отдала прогноз и текущие измерения со станции
  - *Негативный сценарий*: нет прогноза или нет текущего измерения -> отправить администратору сообщение о неисправности, показать ошибку.
  - *Негативный сценарий*: сервер упал -> фронт показал сообщение об ошибке, сообщил оператору.

- **US-2.** Как станция-сенсор, я хочу отправить текущее измерение, чтобы система его обработала и построила прогноз.
  - **Дано**: измерение с приборов. **Когда**: станция пытается отправить измерение. **Тогда**: система приняла измерение и записала его в БД.
  - *Негативный сценарий*: станция отправила пустую/некорректную информацию -> не записываем измерение и отправляет администратору сообщение о неисправности станции-сенсора.
  - *Негативный сценарий*: сервер упал -> фронт показал сообщение об ошибке, сообщил оператору.

- **US-3.** Как пользователь, я хочу посмотреть прогноз, чтобы понять как одеться завтра.
  - **Дано**: существующий прогноз погоды. **Когда**: пользователь заходит на сайт. **Тогда**: сервер отдает прогноз.
  - *Негативный сценарий*: сервер упал -> фронт показал сообщение об ошибке, сообщил оператору.

- **US-4.** Как администратор, я хочу добавить или изменить информацию о метеостанции или поменять интеграцию, чтобы точнее строился прогноз.
  - **Дано**: список метеостанций. **Когда**: администратор вносит изменения. **Тогда**: сервер принимает изменения и вносит их в БД.
  - *Негативный сценарий*: сервер упал -> фронт показал сообщение об ошибке, сообщил оператору.
  - *Негативный сценарий*: добавленный/измененный метеоисточник не отвечает/недоступен -> отказать в добавлении в добавлении.

- **US-5.** Как оператор, я хочу управлять сервером и БД, чтобы поддерживать работоспособность сервиса.
  - **Дано**: сообщение об ошибке сервера/БД. **Когда**: пришло уведомление о сбое. **Тогда**: оператор чинит нужный сервис и перезапускает его.
  - *Негативный сценарий*: нет информации о сбое тк сервер упал -> оператор едет в дата-центр и поднимает его руками.

- **US-6.** Как пресса, я хочу получать валидные сводки о погоде в регионе, чтобы представить информацию аудитории.
  - **Дано**: прогнозы в регионе. **Когда**: пресса запросила сводку. **Тогда**: сервер формирует сводку и отправляет.
  - *Негативный сценарий*: сервер упал -> фронт показал сообщение об ошибке, сообщил оператору.

- **US-6.** Как внешний API-сервис (партнер), я хочу получить сырые прогнозы, чтобы использовать в своих коммерческих целях.
  - **Дано**: прогнозы. **Когда**: партнеру понадобилась информацию. **Тогда**: сервер отправляет нужные прогнозы.
  - *Негативный сценарий*: сервер упал -> фронт показал сообщение об ошибке, сообщил оператору.

## Шаг 3. NFR

| Категория   | Требование            | Число (прикидка) | Откуда число                                |
|-------------|-----------------------|------------------|---------------------------------------------|
| Throughput  | приём телеметрии      | 500 изм/с        | поток сенсоров: 500 станций × 1 измерений/с |
| Latency     | ответ API аналитики   | p99 < 300 мс     | из US про дашборд                           |
| Надёжность  | потери телеметрии     | 0 при отказе ≤ T | факт нельзя терять                          |
| Сохранность | срок хранения истории | 5 лет            | нужно для сравнения прогноз/факт            |

# Шаблон: FR (user stories) и NFR

FR — *что* система делает. NFR — *насколько хорошо*. На Семинаре 1 FR прорабатываем глубоко — через роли и user stories.

---

## Шаг 1. Роли (акторы)

Кто пользуется системой или взаимодействует с ней.

| Роль | Что хочет от системы |
|------|----------------------|
| напр. Аналитик | смотреть ошибку прогноза по станциям |
| Станция-сенсор | сдавать поток измерений факта |
| … | … |

## Шаг 2. User stories

На каждую ключевую функцию — история в формате:

> **Как** `<роль>`, я хочу `<действие>`, чтобы `<ценность>`.

И к ней — критерии приёмки:

> **Дано** `<предусловие>`; **Когда** `<действие>`; **Тогда** `<ожидаемый результат>`.

**Приоритет:** MVP / позже.

### Бланк

- **US-1.** Как …, я хочу …, чтобы … `[MVP]`
  - **Дано** … **Когда** … **Тогда** …
  - *Негативный сценарий* (обязательно): нет данных / сбой приёма / данные протухли → что тогда?

## Шаг 3. NFR

*Насколько хорошо.* Каждому требованию — число и откуда оно взялось.

| Категория | Требование | Число (прикидка) | Откуда число |
|-----------|-----------|------------------|--------------|
| Throughput | приём телеметрии | напр. N изм/с | поток сенсоров: M станций × k измерений/с |
| Latency | ответ API аналитики | напр. p99 < X мс | из US про дашборд |
| Надёжность | потери телеметрии | напр. 0 при отказе ≤ T | факт нельзя терять |
| Сохранность | срок хранения истории | напр. Y лет | нужно для сравнения прогноз/факт |

> **Связка:** хотя бы один NFR должен **вытекать из user story** — ожидание из истории превращается в число NFR.

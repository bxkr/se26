# ADR-002: Тип хранилища для исторических измерений

**Статус:** Принят  
**Дата:** 29.06.2026  
**Связано с требованием:** NFR throughput (500 изм/с на запись), NFR latency (p99 < 300 мс для API аналитики), NFR сохранность (хранение 5 лет), US-1 «Аналитик смотрит ошибку прогноза по станции за период».

### Контекст

Основной паттерн записи — непрерывный поток 500 изм/с, всегда в «конец» по времени. Основной паттерн чтения — выборка по одной станции за произвольный период (US-1). За 5 лет при 500 изм/с накапливается ~79 млрд строк; обычная реляционная таблица без специализированного сжатия и индексов по времени не выдержит ни по объёму, ни по latency.

### Решение

Используем TimescaleDB (расширение PostgreSQL) с гипертаблицей `measurements`, партиционированной по времени. Индекс `(station_id, time)` покрывает основной запрос аналитики; автоматическое сжатие чанков снижает объём.

### Рассмотренные альтернативы

- **PostgreSQL без расширений** — отвергнуто: без партиционирования по времени и специализированного сжатия запросы по диапазону дат на 79 млрд строк не уложатся в p99 < 300 мс; объём на диске неприемлем.
- **InfluxDB** — отвергнуто: нет SQL, сложнее джойнить с таблицей прогнозов при вычислении ошибки (US-1); меньше зрелость экосистемы для транзакционных операций.
- **Cassandra** — отвергнуто: избыточна при 500 изм/с; операционная сложность высокая; агрегации (fan-out по партиции) неудобны для аналитических запросов US-1.

### Последствия

- **Плюсы:** партиционирование по времени ускоряет range-запросы; сжатие чанков снижает объём хранения в 10–20×; SQL-совместимость упрощает JOIN с таблицей прогнозов.
- **Минусы / цена:** зависимость от расширения PostgreSQL; команда должна знать специфику TimescaleDB (политики сжатия, retention); миграция данных при смене движка сложнее, чем с ванильным Postgres.
